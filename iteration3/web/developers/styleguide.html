Code Style Guidelines

Arithmetic operaters should be spaced to increase clarity but can be left close together if it makes order of operations clearer

// Well spaced for readability
if (i + j == (size - 1)) filter[i][j] = 1;
ColorData(a.m_red + b.m_red, a.m_green + b.m_green, a.m_blue + b.m_blue, a.m_alpha + b.m_alpha);
val = m_settings.coeff[2] * dist_cent * dist_cent + m_settings.coeff[1] * dist_cent + m_settings.coeff[0];
// Multiplication operator left tight to m_size to make division first clear
int app_x = x - (m_size)/2;
int app_y = y + (m_size)/2;

Minimization of depth is preferred, move extensive nesting into helper functions

Avoid reptitive code that could be moved into a function

Code should be documented where necessary but variables names should be chosen to reduce need for documentation

Commented out sections of code should have an explaination for their removal and continued presence

Debugging output should be removed from library functions

Dependencies should be minimized to what is used in the file

Mask does not depend on any other parts of the library

Classes should have their own file unless they are small enough that adding a new file would decrease readability

Tool classes are quite small and are left in the same file, but will likely be split off if many more tool types are added

Indentation size should be 4 and use spaces

Class inheritance should be used preferentially especially in cases where a parent class already exists

Constants should be defined prior to use

Static functions should be placed in a properly named namespace

Class names should be descriptive and have capital lettering consistent with the rest of the library

Allocated memory should be deleted. Functions that allocate memory and do not delete it should have a comment alerting the user that they must cleanup the memory

Source files should be placed in the src folder

Header files should be used for all cpp files, but the implementation of functions inline in header files should be avoided

Code documentation is used to generate html docs for the library. Follow Doxygen comment standards for all major classes and functions
