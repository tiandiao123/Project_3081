Blog Entry 3: Testing filters

This section of code is pulled from the TestManager class which I wrote most of. I also spent extensive time refactoring and docuenting codecode and setting up the library including all the make files.

int TestManager::testFilters(int nStart, int nEnd) {
    int res = 0;
    res += testFilters_format(nStart, nEnd, JPG_EXT);
    res += testFilters_format(nStart, nEnd, PNG_EXT);
    return res;
}

int TestManager::testFilters_format(int nStart, int nEnd, std::string format) {
    int res = 0;
    for (int i = nStart; i < nEnd; ++i) {
        app_fp->setImageFile(BASE_IMG_PATH_NOEXT + "." + format);
        app_fp->loadImageToCanvas();
        runFilter(i);
        app_fp->setImageFile(FILT_TEST_PATH_NOEXT + itos(i) + "." + format);
        app_fp->saveCanvasToFile();
        res += compareFilesByPixelBuffer(FILT_TEST_PATH_NOEXT + itos(i) + "." + format, FILT_RES_PATH_NOEXT + itos(i) + "." + format);
    }
    return res;
}

int TestManager::runFilter(int idx) {
    int init_idx = app_fp->UI_APPLY_BLUR;
    idx += init_idx;
    app_fp->gluiControl(idx);
}

The purpose of this code is to test the FlashPhoto filters sequentially on both jpegs and pngs which consequently tests image saving and loading as well. The code is designed to test a range of filters such that all, several, or a single filter can be run. testFilters delegates to a helper function testFilter_format which takes an additional format argument. This means that code does not need to be replicated for the testing of each format. testFilters also initializes a result integer which is added between the two testFilter_fomat calls. This value will be zero if neither function errors or nonzero if tests fail in either function.

testFilter_format loops through the range of filters to apply adding the result of each run to a result variable as in the previous function. Here, the first reference to app_fp is found. app_fp is a pointer to an instance of FlashPhotoApp. As many functions as possible are run as though FlashPhoto was receiving commands from the user to simulate realistic input. First, FlashPhotos image path pointer is set to a base image used for all tests. Then, the image is loaded to the canvas and a filter is run with runFilter. The image path pointer is set to a test result path with an integer identifier appended and the canvas is saved to the file. The result variable is then updated by comparing just saved file with a premade file of expected output.

runFilter uses a trick to press the filter button by pulling the first GUI id for filter buttons from the GLUI control enumeration then adding the filter index to the value to get the index of the desired filter button. gluiControl is then used to call function. This was exceptionally easier than adding a large switch statement and more future proof as more filters are added to FlashPhoto. However, this is a tradeoff in readability and has some limitations. For example, new filters UI control ids need to be placed sequentially after the other Filter ids.

Notice that all the paths used are defined in the tests file for easy changes. Additionally, variables and functions are well named such that very little explaination is needed to understanding what is happening.

Note: I didn't write a lot of new, complex code this release because I didn't work on Mia. Hopefully my extra long last blog entry makes up for it !

Blog Entry 2: PixelBufferHistory

This section of code is pulled from the PixelBufferHistory class which I wrote all of. I also did extensive work on all aspects of the iteration such as the design of all of the Filter class structure and the blur tool.

void PixelBufferHistory::extend() {
    std::cout << "Extending history stack..." << std::endl;
    int n_size = m_size + EXTEND_SHRINK_SIZE;
    int n_off = 0;
    // Check for max size and setup rollover
    if (n_size > MAX_HISTORY_SIZE) {
        std::cout << "Reached maximum size, dropping beginning items..." << std::endl;
        n_size = MAX_HISTORY_SIZE;
        n_off = EXTEND_SHRINK_SIZE;
        m_filledsize -= EXTEND_SHRINK_SIZE;
        m_pos -= EXTEND_SHRINK_SIZE;
    }
    // Allocate and copy
    PixelBuffer **n_data = new PixelBuffer*[n_size];
    for (int i, j = 0; i < n_size && j < m_size; j = ++i + n_off)
        n_data[i] = m_data[j];
    // Cleanup old
    if (m_data) delete [] m_data;
    // Update pointer and size
    m_data = n_data;
    m_size = n_size;
}

void PixelBufferHistory::shrink() {
    std::cout << "Shrinking history stack..." << std::endl;
    int n_size = m_size - EXTEND_SHRINK_SIZE;
    // Sanity check
    if (n_size < m_pos || n_size < EXTEND_SHRINK_SIZE || n_size < EXTEND_SHRINK_SIZE) {
        std::cout << "Stack unready to shrink" << std::endl;
        return;
    }
    // Allocate and copy
    PixelBuffer **n_data = new PixelBuffer*[n_size];
    for (int i = 0; i < n_size; ++i)
        n_data[i] = m_data[i];
    // Cleanup old
    for (int i = n_size; i <= m_filledsize; ++i)
        if (m_data[i]) delete m_data[i];
    delete [] m_data;
    // Update pointer and sizes
    m_data = n_data;
    m_size = n_size;
    m_filledsize = m_size - 1;
}

The purpose of this code is manage the size of the stack that pixel buffer history used for undo-redo is stored in. These functions are called from push and pop respectively if the criteria are met. Extend is called when the stack is full, in other words, m_size (the size of the array) = m_filledsize (the number of items in the stack). Shrink is called when a large amount of items have been popped such that m_pos (the current index pointed to in the stack) + SHRINK_REQ (some constant) < m_size. These functions work together to ensure that the array of PixelBuffers in PixelBufferHistory does not use too much memory.

Extend declares several important values that are used for the creation of a new array. This is an expensive operation so the new size (n_size) is expanded by a constant (EXTEND_SHRINK_SIZE) which I have set to 10 but could be expanded if a lot of operations are being sent to the history. An integer offset (n_off) is declared for use if the maximum history size is reached. This check is essential for preventing an ever expanding amount of memory use and to handle it all we need to do is change some values before the copy operation. We reset n_size to the maximum size, set n_off to the number of PixelBuffers we will remove from the bottom of the stack, shrink the filled size of the stack to reflect those removed items, and move the stack pointer down to reflect the removed items (these last two changes would make more sense happening after the old stack array is copied to the new one but would require another if statement and these changes do not affect the copying operation so its done here).

The next section of Extend copies the old array into the new array. A new array of PixelBuffer pointers is allocated and a loop begins with i representing the position in the new array and j representing the position of the item to copy from the old. Notice that i == j if we are not at the maximum stack size since n_off == 0. These assignments have been condensed into the header of the for loop which sacrifices some clarity for compactness. Arguably, the loss of clarity is minimal; compare the current version to several of the old iterations:

(a brief note for those who dont know the difference between ++i and i++ which is essential here, ++i updates the value of i then returns the new value, i++ updates the value of i and returns the old value)

// Initial
for (int i = 0; i < m_size; ++i) {
    int j = i + n_off;
    if (j >= m_size) break;
    n_data[i] = m_data[j];
}

// Second
for (int i, j = 0; i < n_size && j < m_size; i++, j = i + n_off)
        n_data[i] = m_data[j];

// Current
for (int i, j = 0; i < n_size && j < m_size; j = ++i + n_off)
        n_data[i] = m_data[j];

After copying to the new array, we free the pointer to the old array and then update the member variable to point to the new array and update the member variable m_size to reflect the new arrays size.

Shrink functions in a similar manner. The n_size is set by removing the shrink size from the current size. Next, the function checks whether it should be called. This differs from extend() as adding to the array size cannot cause serious problems. Shrinking the array, however, can result in loss of data from the stack, so although it is a private member of PixelBufferHistory and should not be called incorrectly, it is worth making the check. A new array is allocated and copied, but no offset is needed since the shrinking here is always happening on the top of the stack. Cleanup requires more depth here because PixelBuffer objects can be left dangling in the height difference between the new and old stacks. PixelBuffer objects are freed from that section if they exist and m_data is deleted as before. Notice that m_filledsize has to be updated since we deleted items from the stack, the value is n_size - 1 because we iterate from 0(n_size - 1).

I choose these two functions because many of the other snippets were simplified by our design structure and would need to show several sub-functions e.g. FilterHandler apply -> Filter (wrapper) apply -> Outer function applying -> Inner function doing unique work -> Outer function completion. Additionally, this is an extension of a basic data structure that weve learned a lot about so the general concept should be easy to understand allowing the reader to focus on the coding style and benefit of these specific functions. By including two functions, I can show a consistency in style. Since both functions do a similar task (modify the size of the underlying array) the structure of their code should be very similar. Try to maintain this in your own code, find a way that seems to be the simplest expression of what youre trying to accomplish then repeat it when you have to accomplish a similar goal. This allows someone to become used to your code rather than relearning your style at every function.

Notice that I do not use raw constants like m_size + 10, but rather use constants defined as static consts in the PixelBufferHistory.h class definition. This not only results in easier maintenance of the code (changing a value in one place rather than 6), it results in better readibility and clarity. Additionally, it allows easy implementation of modification of those variables per class instance.

I have pretty minimal use of comments here because the variable names are relatively clear and computations are relatively straight forward. Some comments are used to space out distinct goals so the function isnt a block of code doing several tasks without clear distinction. . Make sure you manage your memory well! This certainly gets tricky with dynamically sizing structures. Just remember to match each new with a delete and you should be off to a great start. My initial versions of these functions did not delete m_data before reassigning it to n_data. Be wary when you are declaring an object using new and not deleting it in that same function. Notice as I have multiple extend operations Im cleaning up the n_data that I allocated the previous time. The destructor for the class also takes care of the final m_data array as well as all the PixelBuffers sitting in it.

Blog Entry 1: Blending colors

ColorData Tool::blendColor(ColorData col_cur, ColorData col_new, float opac_new, int mixing_method) {
    // There are several techniques to build new colors
    // and accordingly, I will use several here.
    // Account for error in floats, handle transparent painting quickly
    if (opac_new < 1.0e-6) return col_cur;
    ColorData ret;
    // Select a mixing method and blend to get a new pixel ColorData
    switch (mixing_method) {
        case BASICBLEND:
            col_new = col_new*opac_new;
            col_cur = col_cur*(1-opac_new);
            ret = col_cur + col_new;
            break;
        case PHOTONFLUXBLEND:
            ret.setRed(photonfluxblendColorVal(col_cur.getRed(), col_new.getRed(), opac_new));
            ret.setGreen(photonfluxblendColorVal(col_cur.getGreen(), col_new.getGreen(), opac_new));
            ret.setBlue(photonfluxblendColorVal(col_cur.getBlue(), col_new.getBlue(), opac_new));
            ret.setAlpha(1.0);
            break;
        case LUMINANCEBLEND:
            ret = col_new * opac_new * col_cur.getLuminance()
                + col_cur * (1.0 - opac_new * col_cur.getLuminance());
            break;
        default:
            ret = col_new;
    }
    return ret;
}

The purpose of this code is to blend the color of two color data objects given one curent pixel, the new color being applied, and the opacity of the new color. This code was previously found directly inserted into the function which applies a tool to the PixelBuffer, but was moved into a separate function for several important reasons.

First, theres the addition new color blending methods such as the (I made up the name) photon flux blend which uses more complex mathematics to better determine the final color of the mixed pixel and the luminance blend (which was described in class) which depends on color luminance rather than direct RGB values and is especially useful for the highlighter. These additions if done without movement to their own function would have made the tool application function significantly more convoluted. In general, increased code depth (tabs, loops in loops, a in b in c in d) is related to decreased readibility, and the switch statement neccesary for this function would have drastically increased the depth of color blending which occurs as one of the final steps in the tool application loop (already two for loops and an if statement deep). As an added benefit, the steps found in BASICBLEND are now described as a function blendColor rather than a comment in the middle of many other tasks.

Second, the presence of several tool subclasses that need to blend colors makes the movement of this code into a function extremely helpful. Instead of copying and pasting the code to patch the several spots it needs to be used, the function can be placed in the parent class for the use of all types of tools. This is especially relevant as more tools are added.

Third, future additions of color blending methods are much easier. Instead of finding all uses of code like this and updating them with the new method, a new statement can be added the switch and the ColorMixingMethod enum and the new method is ready to be used by any tool. This allows rapid changing of the blending method tools use and rapid prototyping of new blending techniques.

The code accomplishes this purpose using a switch of the enumeration ColorMixingMethod. Depending on the method it implements a color blending algorithm using percentages of the old and new color as described by opac_new. For example, BASICBLEND takes each color and multiplies it by the needed opacity (the new color gets the opac_new e.g. 0.8 and the old color gets 1-opac_new e.g. 0.2). These colors are then added to create a return color. A more complex method such as PHOTONFLUXBLEND which applies sqrt((1 - opac)(old)^2 + opac*(new)^2) moved computation into a separate function to avoid repetitive writing of the computation.LUMINANCEBLEND is a straightforward implementation of the highlighter color blending technique described in class, broken onto two lines to clarify the differences in how each color is treated. Finally, if an unknown blending technique is passed into the function, it defaults to applying 100% the new color. Likewise, before doing any of these checks, the function checks if the opacity of the new color is transparent at which point it will exit returning the old color saving computation.

Each of these techniques uses a different style of programming which can help elucidate which is clearer. The first alters the variables that are passed in before adding them, each on a separate line, making use of the functions included in ColorData to handle component addition and multiplication. While altering passed variables is dangerous sometimes, in C, as long as they are not pointers, it will not alter the origin. The next sets each each color separately because the functions in ColorData do not support square root, which reduces clarity but is neccessary. The last could easily be implemented as the first was to increase clarity, but since the changes to col_new is dependent on data from col_cur, altering the variables would make the order at which they are executed important and since this is not true of any of the other cases here, I try to avoid it.

Comments here are sparse due to the brevity of the code, but it may be wise to have documentation explaining how each mixing method works and the reasoning behind their mathematics if this was published in a library of code.
